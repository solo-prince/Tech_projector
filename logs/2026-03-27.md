## 🚀 Project Update - Autonomous Warehouse Rover (AWR-v2)

**Date:** 2026-03-27

**Progress:**
The primary objective this sprint was to refine the navigation stack's performance in dynamic environments. We conducted extensive testing of the localization algorithms and path planning logic within our simulated warehouse environment and on the physical hardware. The focus was on improving obstacle avoidance and minimizing localization drift during high-speed maneuvers.

**Technical Work:**
- **Navigation Stack:** Implemented ROS 2 (Humble Hawksbill) Nav2 stack. We primarily focused on tuning the parameters for the DWA (Dynamic Window Approach) local planner within the `move_base` configuration.
- **Odometry Correction:** Integrated IMU data (MPU6055) with wheel encoders to improve odometry estimation. This involved calculating a complementary filter in the `robot_localization` package to reduce accumulated errors from wheel slippage.
- **Path Planning Optimization:** Tuned costmap parameters, specifically adjusting the `inflation_radius` and adding a new layer for dynamic obstacles (e.g., moving carts and people) to improve reactivity and prevent collisions while maintaining efficient path execution.

**Issues & Fixes:**
- **Localization Drift:** Initial testing showed significant drift after 10-15 meters of travel, primarily due to wheel slip on concrete flooring.
- **Fix:** Implemented IMU-based odometry correction as described above. We found a balance where the IMU corrects large errors while the wheel encoders maintain short-term accuracy, reducing overall drift by approximately 60% in high-speed runs.

**Next Steps:**
- **Integration with Mission Planner:** Integrate the refined navigation stack with the high-level mission planning software (which receives tasks from the inventory management system).
- **Multi-Robot Simulation:** Begin simulation testing for multi-robot coordination to ensure collision-free operation when multiple AWR-v2 units are operating simultaneously in the same area.

## 🚀 Project Update - Smart Farm Monitoring System (SFMS)

**Date:** 2026-03-27

**Progress:**
We successfully completed the initial phase of sensor integration and data acquisition for the SFMS prototype. The focus was on establishing reliable data transmission from the sensor nodes (Arduino/ESP32) to the central data broker via MQTT. The system is currently collecting real-time environmental data (temperature, humidity, soil moisture) from three test beds.

**Technical Work:**
- **Hardware Integration:** Developed custom PCBs to simplify wiring for the sensor nodes. Integrated DHT22 (temperature/humidity) and capacitive soil moisture sensors (instead of resistive ones to avoid corrosion issues).
- **IoT Protocol Implementation:** Configured ESP32 microcontrollers to connect to the local Wi-Fi network and publish sensor data to a Mosquitto MQTT broker running on a Raspberry Pi. The data payload format was standardized to JSON for easier parsing by the backend.
- **Code Logic:** Implemented a simple data sampling routine in Arduino C++. To ensure data accuracy, we applied a moving average filter in software to smooth out high-frequency noise from the capacitive soil moisture sensor readings.

**Issues & Fixes:**
- **Wi-Fi Instability:** The ESP32 nodes occasionally lost connection to the Wi-Fi network, leading to gaps in the sensor data log.
- **Fix:** Implemented a robust reconnection logic using the `WiFiManager` library. This allows the device to automatically attempt reconnection multiple times and reset if necessary, improving uptime significantly.

**Next Steps:**
- **Actuator Integration:** Begin work on the output side of the system, integrating relays to control irrigation pumps based on predefined soil moisture thresholds.
- **Dashboard Development:** Create a web-based dashboard using Node-RED to visualize the real-time sensor data and control automation rules.

## 🚀 Project Update - AI Defect Detection for Manufacturing

**Date:** 2026-03-27

**Progress:**
This sprint focused on the initial training and evaluation of the deep learning model for classifying defects on manufactured components. We completed the first full training run on the collected dataset and evaluated the performance metrics against our baseline requirements.

**Technical Work:**
- **Dataset Preparation:** Collected and annotated ~5,000 images of components, categorizing them into four classes: "Good," "Scratch," "Dent," and "Foreign Object." Implemented data augmentation techniques (random rotations, brightness shifts) to expand the dataset and improve model robustness.
- **Model Training:** Used Python with TensorFlow/Keras to train a transfer learning model based on the EfficientNetB0 architecture. The model was trained for 20 epochs using the Adam optimizer with a learning rate of 0.001.
- **Performance Evaluation:** Achieved an initial F1 score of 0.88 across all classes. The model shows strong performance in detecting "Good" and "Scratch" categories but struggles with "Dent" due to visual similarity with normal variations in component texture.

**Issues & Fixes:**
- **Class Imbalance:** The dataset contained significantly fewer examples of "Dent" and "Foreign Object" compared to "Good." This resulted in lower precision and recall for the minority classes.
- **Fix:** Implemented class weighting during training to penalize misclassifications of minority classes more heavily. This improved the F1 score for "Dent" from 0.72 to 0.81 in a subsequent run.

**Next Steps:**
- **Model Quantization:** Optimize the trained model for embedded deployment by converting it to TensorFlow Lite (TFLite) for use on an edge device (e.g., NVIDIA Jetson Nano or Raspberry Pi).
- **Inference Speed Testing:** Measure the inference latency on the target hardware to ensure real-time performance requirements are met for the production line.
