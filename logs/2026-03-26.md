## 🚀 Project Update - AquaFarm Controller

**Date:** 2026-03-26

**Progress:**
The primary objective for this sprint was integrating the pH sensing subsystem and establishing reliable data reporting via MQTT. We successfully installed and calibrated the Atlas Scientific EZO pH module and implemented the necessary code to read and publish data to the local Mosquitto broker. Initial end-to-end testing confirmed data flow from sensor acquisition on the ESP32 to a Grafana dashboard.

**Technical Work:**
*   **Sensor Integration:** Hardware integration of the Atlas Scientific EZO pH module. Configured the ESP32's I2C bus (SDA: GPIO 21, SCL: GPIO 22) for communication with the EZO module in I2C mode.
*   **Firmware Logic:** Developed new C++ classes for the pH sensor driver. Implemented a non-blocking read routine to avoid main loop stalls during sensor acquisition. The code now performs a pH reading every 10 seconds.
*   **Data Serialization:** Implemented JSON serialization using the `ArduinoJson` library for data transmission. The payload format is `{"timestamp": <unix_time>, "ph": <ph_value>, "temp": <temperature_value>}`.
*   **MQTT Client:** Configured the `PubSubClient` library to connect to the local Mosquitto broker. The controller publishes to the topic `aquafarm/sensor/ph_data`.

**Issues & Fixes:**
*   **Issue:** Intermittent "ghost" readings and communication failures with the EZO module. The I2C bus would occasionally drop communication, requiring a full system reboot.
*   **Fix:** Traced the issue to a timing conflict during the sensor read cycle and potential I2C bus contention with another component (the real-time clock). Resolved by implementing a robust error handling routine in the I2C read function, including a timeout and retry mechanism. We also added a hardware fix by verifying external pull-up resistors (4.7kΩ) on the I2C lines to ensure clean signal integrity.

**Next Steps:**
*   **Actuator Integration:** Begin work on the closed-loop control system. Integrate the peristaltic pumps (up/down pH solutions) via relay modules.
*   **Control Logic:** Develop the state machine for pH regulation. Implement logic to automatically dispense solutions based on predefined setpoints (e.g., maintain pH between 5.8 and 6.2).
*   **Dashboard Enhancements:** Finalize the Grafana dashboard to include historical data, real-time gauges, and alerts for out-of-range pH values.

## 🚀 Project Update - Autonomous Rover SLAM Navigation

**Date:** 2026-03-26

**Progress:**
This update focused on integrating the 2D LIDAR sensor and configuring the initial Simultaneous Localization and Mapping (SLAM) stack using ROS 2 (Humble Hawksbill). We successfully completed the mapping phase of our test environment (an empty warehouse bay), generating a reliable occupancy grid. The rover can now create a map in real-time while being teleoperated.

**Technical Work:**
*   **LIDAR Integration:** Mounted the RPLIDAR A2M8 sensor on the rover platform and configured the ROS 2 driver (`rplidar_ros`) to publish `LaserScan` data.
*   **SLAM Stack Configuration:** Configured the `slam_toolbox` package for mapping. Adjusted parameters for scan matching, including `max_laser_range` and `scan_buffer_size`, to optimize performance based on the specific environment characteristics.
*   **TF Tree Setup:** Defined the required `tf` transformations for the rover, specifically `base_link` to `laser_link` and `odom` to `base_link`. Verified the transformation tree using `rqt_tf_tree`.
*   **Odom Source:** Integrated wheel encoder data from the motor drivers (via serial communication with the embedded microcontroller) and published it as `nav_msgs/Odometry` messages to provide a robust odometry source for the SLAM algorithm.

**Issues & Fixes:**
*   **Issue:** Initial map generation exhibited significant "drift" and warping, particularly when the rover made sharp turns. The generated occupancy grid showed inconsistent wall locations.
*   **Fix:** Identified the source of the drift as an inaccurate odometry source from the wheel encoders. The encoder data was noisy, leading to errors in position estimation during high-speed turns. Implemented a Kalman filter on the odometry data to smooth position and orientation estimates before feeding them to the SLAM algorithm. Also, increased the `transform_tolerance` parameter in `slam_toolbox` to better accommodate timing differences between the LIDAR scans and odometry updates.

**Next Steps:**
*   **Navigation Stack Integration:** Implement the `Nav2` stack for autonomous navigation. This includes configuring `AMCL` (Adaptive Monte Carlo Localization) for localization within the generated map and setting up the planner (e.g., `Dijkstra` or `A*`) for pathfinding.
*   **Obstacle Avoidance:** Integrate the ultrasonic sensors and depth camera data into the local costmap configuration of `Nav2` to enable dynamic obstacle avoidance.
*   **Waypoints:** Develop a mission planner interface to define waypoints for autonomous navigation tasks.

## 🚀 Project Update - AI Defect Detection Model

**Date:** 2026-03-26

**Progress:**
This update focused on iterating on our object detection model for identifying surface defects on manufactured components. We completed the initial training of a new model architecture (YOLOv8) on an expanded dataset. The initial validation results show improved precision and recall compared to the previous version (YOLOv5), specifically in detecting smaller defects (pinholes) and distinguishing between different defect types (cracks vs. scratches).

**Technical Work:**
*   **Dataset Expansion:** Added 500 new images to the dataset, focusing on edge cases and difficult-to-detect defects. Completed annotation using Label Studio for both bounding boxes and segmentation masks.
*   **Model Architecture:** Switched from YOLOv5 to YOLOv8 for improved performance and efficiency. Configured the model with a custom head to classify three distinct defect types: `pitting`, `scratch`, and `crack`.
*   **Training Configuration:** Performed training on a dedicated GPU cluster (NVIDIA A100). Adjusted hyperparameters for a 100-epoch training run. Implemented data augmentation techniques (random rotation, scaling, and HSV adjustments) to increase dataset robustness and mitigate overfitting.
*   **Evaluation:** Conducted validation on a held-out test set. Achieved mAP@0.5 of 89.2% for the `scratch` class and 82.1% for the `pitting` class.

**Issues & Fixes:**
*   **Issue:** Severe class imbalance, particularly for the `crack` defect type, which comprised only 5% of the total annotations. The model consistently failed to detect cracks, resulting in high false negatives for this class.
*   **Fix:** Implemented several strategies to address the imbalance:
    1.  **Weighted Loss Function:** Adjusted the loss function to apply higher weights to the underrepresented `crack` class.
    2.  **Over-sampling:** Applied over-sampling during training specifically for the `crack` class by duplicating images containing this defect.
    3.  **Data Augmentation:** Used advanced augmentation (e.g., `copy-paste`) to introduce synthetic variations of the `crack` defect.

**Next Steps:**
*   **Model Deployment:** Quantize the trained YOLOv8 model for deployment on an edge device (Jetson Nano). Convert the model to ONNX format for optimized inference.
*   **Integration Testing:** Integrate the model into the production environment (C++ inference engine) and perform real-time testing on the production line cameras.
*   **Feedback Loop:** Establish a feedback loop to capture false positives and false negatives from the production line for future dataset iterations.
